# Issue Idea — 2026-03-15

Title: Add `signMessage` support to all wallet drivers for SIWE and sign-in authentication flows

Summary:
Many dApps authenticate users by asking them to sign a plain-text message (e.g., Sign-In With Ethereum / SIWE, or Polkadot's equivalent `signRaw`) rather than submit an on-chain transaction. Chroma currently only exposes `approveTx`/`rejectTx` methods, which target transaction-confirmation popups — but message-signing popups have completely different UI flows in every supported wallet, leaving test authors unable to automate sign-in workflows end-to-end.

Why it matters:

* SIWE (`eth_sign`, `personal_sign`, `eth_signTypedData`) is the de-facto authentication mechanism for Ethereum dApps; without `signMessage`, any login-gated E2E test flow must be skipped or manually stepped through.
* Polkadot dApps (e.g., Hydration, Turtle) similarly trigger `signRaw` message-signing popups that differ from the transaction approval UI, meaning `approvePolkadotJSTx` and `approveTalismanTx` do not match those selectors and will time out.
* Adding this capability unlocks full end-to-end coverage of the most common real-user interaction pattern in web3 — connect wallet → sign message → access authenticated route — for all three supported wallets in a single release.

Suggested scope:

* Add `approveSignMessage(page)` and `rejectSignMessage(page)` to `src/wallets/metamask.ts`, targeting MetaMask's "Sign this message" side-panel UI (distinct `data-testid` attributes from the "Confirm" transaction panel).
* Add `approveSignMessage(page)` and `rejectSignMessage(page)` to `src/wallets/polkadot-js.ts`, targeting the extension's `signRaw` popup (the "Sign message" heading + password field flow used for raw bytes signing).
* Add `approveSignMessage(page)` and `rejectSignMessage(page)` to `src/wallets/talisman.ts` and expose all six new methods via the corresponding factories in `wallet-factory.ts`, so they appear on the typed `wallets['metamask']`, `wallets['polkadot-js']`, and `wallets.talisman` fixture objects.

Category:
feature

Optional notes:
MetaMask distinguishes between `personal_sign` (human-readable message) and `eth_signTypedData_v4` (EIP-712 structured data); both ultimately show the same "Sign" button in Flask v13 but the page content differs. Initially supporting `personal_sign` is sufficient — typed-data support can follow in a separate issue. For Talisman, a `dontShowAgain` checkbox sometimes appears on the message-signing popup; the implementation should handle that case gracefully, consistent with how `approveTalismanTx` already handles the optional "Yes" pre-confirm step.
